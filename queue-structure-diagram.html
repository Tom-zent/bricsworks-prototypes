전자신고 큐 — 동작 구조 정리

근거: [P11-PLN]20260722 전자신고 큐 PRD · 한 건이 흘러가는 순서 → 대기열이 실제로 도는 모습 → 상태값 전이

1한 건의 여정 — 파일 제작부터 완료까지

담당자 입장에서 신고 한 건이 흘러가는 순서. 핵심은 계정은 ①에서 이미 정해지고, 줄은 ②에서 서기 시작한다는 것.

1

파일 제작 — 작성 단계에서 재계산 실행

재계산을 실행하면 전자파일(국세 txt · 지방세 C10~C40)이 생성된다. 이 순간 시스템이 가장 한가한 홈택스 계정을 골라 파일 안에 새긴다(이후 변경 불가). 파일은 DB에 저장만 되고, 아직 대기열에는 들어가지 않는다.

홈택스 미접속 단계 — 여러 건 동시 제작해도 충돌 없음 계정 배정 기준: 계정별 대기+진행 건수 최소
2

담당자 요청 — 버튼을 누르는 순간 줄을 선다

담당자가 검증하기 또는 전자파일 제출을 누르면 그 건이 대기열에 적재된다 (상태: 대기). 두 버튼은 분리된 액션이고, 제출은 검증을 통과한 건만 누를 수 있다(대기열이 아니라 버튼 단에서 막음).

적재 후엔 자리를 비워도 됨 — 워커가 알아서 처리 (비동기) 검증 성공 ≠ 제출 완료 — 제출은 별도로 다시 요청해야 함
3

워커 처리 — 빈 워커가 "락 안 걸린 계정"의 건을 집어간다

워커(기본 5개, 설정으로 확장)는 특정 계정 전담이 아니다. 대기열을 앞에서부터 훑다가 지금 락이 안 걸린 계정의 건을 만나면 집어간다. 집는 순간 그 계정에 락 🔒이 걸리고 (상태: 진행중), 같은 계정의 나머지 건들은 락이 풀릴 때까지 대기열에 그대로 남는다. 계정이 다른 건들끼리는 동시에 병렬로 돈다.

홈택스 연동은 폴링 — 완료 응답이 올 때까지 반복 확인
4

완료 — 락 반납, 다음 건 진행

검증은 홈택스 완료 응답 수신이 기준, 제출은 접수번호를 받아야만 성공이다. 성공이든 실패든 끝나면 락은 반드시 반납된다 — 정상 완료 / 무응답·유실 / 타임아웃 / 실패 4가지 경우 모두. 락이 풀리면 그 계정의 다음 대기 건이 이어서 진행된다.

2대기열이 실제로 도는 모습 — 계정 2개(A·B), 5건 적재 예시

어느 한 순간을 캡처한 그림. 같은 "대기"라도 이유가 두 가지라는 점이 이 구조의 핵심이다.

대기열 (도착순)

①계정A 검증진행중
→ 워커1이 집어감 · 이 순간 계정A에 락 🔒
②계정A 제출대기
순번은 ③보다 앞이지만 계정A가 락 상태 → 락이 풀릴 때까지 대기
③계정B 검증진행중
→ 워커2가 ②를 건너뛰고 집어감 (계정B는 락 없음) · 계정B 락 🔒
④계정A 검증대기
계정A 락 대기 — ①이 끝나면 ② → ④ 순서로 진행
⑤계정B 제출대기
계정B 락 대기 — ③이 끝나야 진행

워커풀 (기본 5개)

워커 1 ① 계정A · 검증 처리 중 A 락 🔒
워커 2 ③ 계정B · 검증 처리 중 B 락 🔒
워커 3 유휴 — 집을 수 있는 건 없음
워커 4 유휴 — 남은 건 전부 락 걸린 계정
워커 5 유휴
"대기"의 두 가지 이유

1) 워커 배정 전 대기 — 방금 적재돼 아직 아무 워커도 안 집은 상태.
2) 락 대기 — 워커는 놀고 있어도, 같은 계정의 앞 건이 진행 중이라 집을 수 없는 상태 (위의 ②·④·⑤).
상태값은 둘 다 "대기"로 같고, 화면 순번도 도착순 그대로 표기된다. 그래서 내 순번보다 뒷 건이 먼저 처리되는 것은 정상 동작이다.

락이 풀리는 순간

①이 완료되면 → A 락 반납 → 유휴 워커가 곧바로 ②를 집어감 (같은 계정 내에서는 먼저 온 순서대로). 동시 처리량은 결국 계정 수가 상한이고, 워커 수는 그걸 받쳐주는 일손이다 — 계정보다 워커가 많으면 남는 워커는 놀고, 적으면 전체가 느려진다.

3상태값 전이 — 대기 / 진행중 / 성공 / 실패 / 종결

기본 흐름은 한 줄이고, 갈림길은 아래 세 가지뿐이다.

대기 ⟶워커 획득 · 계정 락 진행중 ⟶처리 종료 · 락 반납 성공 / 실패
성공이 제출(접수번호 수신)이면 ⟶종결 ※ 종결 후 재작업 시 취하서 절차는 미확정 (리스크 B)
성공이 검증(완료 응답)이면 — 끝이 아니다. 담당자가 제출 버튼을 눌러 ⟶대기 로 재진입 (자동으로 이어지지 않음)
실패는 사유 기록 후 수동 재시도 — 앞단계 수정 → 파일 재제작 → ⟶대기 재적재 · 딜ID로 동일 건 중복 식별 (자동 재시도 없음)
대기중일 때만 ⟶취소 (상태값 변경) — 진행중부터는 중단 불가

4자주 헷갈리는 포인트

워커 5개면 5건이 동시에 돌아가나?
계정 수만큼만 동시에 돈다. 계정이 2개면 워커가 5개여도 동시 처리는 2건이 최대다. 계정이 늘면 워커 수 설정만 올리면 된다(구조 변경 불필요).
내 건보다 늦게 올라온 건이 먼저 끝났다. 버그인가?
정상이다. 화면 순번은 도착순 표기일 뿐, 실제 처리는 계정 락 상황을 따라간다. 같은 계정 안에서는 반드시 먼저 온 순서대로 처리된다.
검증하기 눌러놓고 퇴근해도 되나?
된다. 비동기 구조라 워커가 정해진 시간까지 알아서 처리한다. 단, 검증 성공해도 제출은 자동으로 이어지지 않으므로 제출은 별도로 요청해야 한다.
잘못 올린 건 어떻게 하나?
대기 중이면 취소 가능. 진행중이면 끝날 때까지 중단 불가. 값을 고치려면 앞단계 수정 → 파일 재제작 → 재적재이며, 딜ID로 동일 건임을 식별한다. 진행 중 건의 앞단계 수정을 어디부터 막을지는 미확정(리스크 A).
